Worker Pool &
Scheduling Policies.

The worker pool dispatches GEMM tasks through a pluggable SchedulingPolicy. Three scheduling policies ship today (round-robin, greedy, load-balanced). Two virtual-time alignment strategies — EARLIEST and LATEST — answer different research questions.

Worker pool model

  • WorkerBase → XtGemmWorker (GPU, cublasXt) and CpuWorker (MKL).
  • WorkerPool dispatches via pluggable SchedulingPolicy (round-robin, greedy, load-balanced).
  • Task queue: std::mutex + std::condition_variable, with atomic task counts.
  • Dual-path GPU/CPU with identical public interfaces (AddTask, EnqueueGemm).

Virtual-time alignment: EARLIEST vs LATEST

Two strategies align virtual time across devices:

  • EARLIEST: align all devices to the earliest vt_start (fastest device).
  • LATEST: align all devices to the latest vt_start (slowest device).
DimensionEARLIESTLATEST
Baseline definitionmin(vt_start)max(vt_start)
Baseline deviceFastestSlowest
Time shift directionBackward (smaller timestamps)Forward (larger timestamps)
Typical shift sizeSmaller (conservative)Larger (aggressive)

Worked example: GEMM 0

Original data (3 devices, single GEMM):

Device 0: vt_start = 2,326,719 us  (slowest)
Device 1: vt_start = 2,298,721 us
Device 2: vt_start = 2,215,361 us  (fastest)

Spread = 111,358 us

EARLIEST (baseline = 2,215,361 us):

Device 0: 2,326,719 - 111,358 = 2,215,361 us
Device 1: 2,298,721 -  83,360 = 2,215,361 us
Device 2: 2,215,361 +       0 = 2,215,361 us

LATEST (baseline = 2,326,719 us):

Device 0: 2,326,719 +       0 = 2,326,719 us
Device 1: 2,298,721 +  27,998 = 2,326,719 us
Device 2: 2,215,361 + 111,358 = 2,326,719 us

Which strategy should you use?

Performance comparison → EARLIEST

  • Aligns to the fastest device.
  • Best for comparing per-device compute time fairly.
  • Preserves relative ordering and uses a conservative shift.

Worst-case / SLA analysis → LATEST

  • Aligns to the slowest device.
  • Highlights tail latency and bottlenecks.
  • Useful for reliability and capacity planning.

Cross-experiment comparison → neither

  • Both alter absolute timestamps.
  • Keep original logs and compare deltas, or use a fixed baseline device/time for all experiments.

Recommended workflow

If you also care about worst-case impact, generate both and compare:

python3 scripts/sync_virtual_time.py perf_merged.log \
  --method earliest --output synced_earliest.log

python3 scripts/sync_virtual_time.py perf_merged.log \
  --method latest --output synced_latest.log

python3 scripts/analyze_sync.py perf_merged.log synced_earliest.log \
  > analysis_earliest.txt

python3 scripts/analyze_sync.py perf_merged.log synced_latest.log \
  > analysis_latest.txt
From the EdgeSys '26 measurement set (97 GEMM operations): mean virtual-time shift was 109,763 μs (≈ 110 ms), with the spread ranging from 0 μs to 123,886 μs.
